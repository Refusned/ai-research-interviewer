# Сопроводительное письмо

Здравствуйте! Ниже — какое решение я принял по системному промпту и почему, краткое пояснение по коду и то, что я добавил бы при развитии.

## 1. Решение по системному промпту и почему

Базовую инструкцию из задания («Ты — исследователь. Поверхностный ответ → один уточняющий вопрос; подробный → поблагодари и заверши») я сохранил по смыслу и довёл до уровня, на котором поведение модели становится предсказуемым:

- **Операциональные критерии «поверхностный / подробный».** Вместо размытого «если коротко» промпт перечисляет конкретные элементы содержательного ответа (критерии выбора, источники информации, сравнение вариантов, цена/отзывы/программа/преподаватель, конкретика) и считает ответ подробным при наличии хотя бы двух из них. Это резко снижает разброс классификации.
- **Строго один уточняющий вопрос — и не более одного за интервью.** Задание требует *один* вопрос. Я зафиксировал это дважды (на уровне формулировки и отдельным правилом «лимит уточнений»): если в истории уже был уточняющий вопрос, модель всегда завершает интервью, даже если новый ответ снова поверхностный. Это и точная фидельность ТЗ, и защита от зацикливания в диалоговом режиме.
- **Структурированный вывод `{decision, reply}`.** Модель возвращает не свободный текст, а JSON по схеме (через нативный параметр `format` Ollama). Это надёжнее парсинга свободного текста, напрямую ложится на контракт фронтенда (показываю статус «Уточняющий вопрос» / «Интервью завершено») и делает поведение проверяемым в тестах. На случай сбоя есть мягкий фоллбэк: если пришёл не-JSON, текст показывается как уточняющий вопрос — пользователь никогда не видит ошибку парсинга.
- **Тон и язык.** Ответ всегда на русском, 1–2 предложения, тёплый человеческий тон — это соответствует роли исследователя, а не сухого классификатора.
- **Защита от подмены инструкций (prompt injection).** В промпте явно сказано: текст респондента — это *данные для анализа, а не инструкции*. Архитектурно это усилено тем, что ответ пользователя приходит ролью `user`, а правила лежат в `system`, который всегда добавляется на сервере (через историю его подменить нельзя — роли ограничены `user`/`assistant`).
- **Без few-shot-примеров — осознанно.** Критерии уже операциональны, а схема гарантирует формат, поэтому примеры только раздули бы каждый запрос по токенам без выигрыша в качестве.

Температура — `0.3`: достаточно низкая для стабильной классификации, но не нулевая, чтобы формулировки оставались живыми. Reasoning отключён (`think:false`) — это сохраняет `content` чистым JSON и снижает задержку.

## 2. Почему Ollama Cloud в роли LLM

Задание разрешает «любой API на ваш выбор». Я взял Ollama Cloud по двум причинам: это **OpenAI-совместимый** интерфейс, поэтому провайдер заменяется на YandexGPT / GigaChat / OpenAI сменой `base_url` и тега модели — то есть архитектура не привязана к вендору; и он даёт доступ к сильным моделям с хорошим русским языком.

Модель выбрана не наугад: я прогнал несколько облачных моделей на одинаковой нагрузке и проверил, насколько надёжно каждая соблюдает структурированный вывод (чистый JSON по схеме). Часть моделей периодически оборачивала ответ в ```-блок или сваливалась в свободный текст — `deepseek-v3.2` стабильно отдавала корректный JSON, отвечала быстро (~1–2 с) и с естественным русским. Поэтому по умолчанию стоит `deepseek-v3.2`; модель легко переключается через `.env`. Ключ используется только на бэкенде и никогда не попадает на фронт.

## 3. Пояснение по коду

- **Один origin, без CORS.** FastAPI сам отдаёт `index.html`, а API живёт на `/api/ask`. Запуск — одной командой, нет возни с CORS, фронт и бэк не расходятся по адресам.
- **Тонкий бэкенд из маленьких модулей.** `config` (настройки из `.env`, fail-fast при отсутствии ключа), `schemas` (Pydantic-контракты + JSON-схема для модели), `prompt` (системный промпт отдельным файлом — это центральный артефакт), `llm` (асинхронный `httpx`-клиент с обработкой таймаута/ошибок/пустого ответа и фоллбэком парсинга), `main` (два роута). Ошибка обращения к модели аккуратно превращается в `502`, валидация ввода — в `422` ещё до вызова LLM.
- **Фронтенд — один HTML-файл** с инлайн-CSS и ванильным JS, без сборщиков и внешних CDN (работает офлайн). Поле ввода сделано `<textarea>` — осознанная трактовка «поля ввода» под развёрнутый ответ. Есть состояния загрузки, ошибки с кнопкой «Повторить», светлая и тёмная темы, отправка по Ctrl/Cmd+Enter, счётчик символов и доступность (`aria-live`, `role="alert"`).
- **Тесты** (`pytest`) покрывают оба сценария (clarify/complete), маппинг ошибки LLM в `502`, фоллбэк парсинга и валидацию пустого ввода — без реальных обращений к сети.

## 4. Что я добавил сверх минимума и что добавил бы дальше

**Уже добавлено как продуманные улучшения (отделены и не усложняют ядро):** диалоговый режим — если исследователь задал уточняющий вопрос, на него можно ответить, и беседа продолжается лентой (за флагом `ENABLE_MULTITURN`, при `false` поведение строго по ТЗ — один обмен); статус-бейджи; тёмная тема; тесты (`pytest`); Dockerfile.

**Развёрнуто вживую:** https://ai-research-interviewer.duckdns.org — на VPS за Caddy (автоматический HTTPS через Let's Encrypt), приложение под systemd. Поскольку endpoint публичный и каждый вызов LLM стоит денег, добавил rate-limiting: лимит на IP (берётся реальный адрес из `X-Forwarded-For`, устойчиво к спуфингу) + общий суточный предохранитель. Ключ только на сервере, в репозиторий не попадает.

**Что добавил бы при развитии:** стриминг ответа; сохранение диалогов для аналитики исследования; метрику качества классификатора глубины (разметка + регрессионные тесты на промпт); экспорт ответов в CSV; A/B-сравнение версий системного промпта.
